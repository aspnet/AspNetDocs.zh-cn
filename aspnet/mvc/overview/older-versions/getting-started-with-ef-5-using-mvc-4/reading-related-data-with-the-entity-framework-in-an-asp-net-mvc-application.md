## <a name="lazy-eager-and-explicit-loading-of-related-data"></a><span data-ttu-id="839f1-116">延迟、 及早计算，和显式加载相关数据</span><span class="sxs-lookup"><span data-stu-id="839f1-116">Lazy, Eager, and Explicit Loading of Related Data</span></span>

<span data-ttu-id="839f1-117">有几种方法，实体框架可以相关的数据加载到实体的导航属性：</span><span class="sxs-lookup"><span data-stu-id="839f1-117">There are several ways that the Entity Framework can load related data into the navigation properties of an entity:</span></span>

- <span data-ttu-id="839f1-118">*延迟加载*。</span><span class="sxs-lookup"><span data-stu-id="839f1-118">*Lazy loading*.</span></span> <span data-ttu-id="839f1-119">首次读取实体时，不检索相关数据。</span><span class="sxs-lookup"><span data-stu-id="839f1-119">When the entity is first read, related data isn't retrieved.</span></span> <span data-ttu-id="839f1-120">然而，首次尝试访问导航属性时，会自动检索导航属性所需的数据。</span><span class="sxs-lookup"><span data-stu-id="839f1-120">However, the first time you attempt to access a navigation property, the data required for that navigation property is automatically retrieved.</span></span> <span data-ttu-id="839f1-121">这会导致发送到数据库的多个查询 — 一个用于实体本身，一个必须检索每个相关实体数据的时间。</span><span class="sxs-lookup"><span data-stu-id="839f1-121">This results in multiple queries sent to the database — one for the entity itself and one each time that related data for the entity must be retrieved.</span></span> 

    ![Lazy_loading_example](https://asp.net/media/2577850/Windows-Live-Writer_Reading-Re.NET-MVC-Application-5-of-10h1_ADC3_Lazy_loading_example_2c44eabb-5fd3-485a-837d-8e3d053f2c0c.png)
- <span data-ttu-id="839f1-123">*预先加载*。</span><span class="sxs-lookup"><span data-stu-id="839f1-123">*Eager loading*.</span></span> <span data-ttu-id="839f1-124">读取该实体时，会同时检索相关数据。</span><span class="sxs-lookup"><span data-stu-id="839f1-124">When the entity is read, related data is retrieved along with it.</span></span> <span data-ttu-id="839f1-125">此时通常会出现单一联接查询，检索所有必需数据。</span><span class="sxs-lookup"><span data-stu-id="839f1-125">This typically results in a single join query that retrieves all of the data that's needed.</span></span> <span data-ttu-id="839f1-126">使用指定预先加载`Include`方法。</span><span class="sxs-lookup"><span data-stu-id="839f1-126">You specify eager loading by using the `Include` method.</span></span>

    ![Eager_loading_example](https://asp.net/media/2577856/Windows-Live-Writer_Reading-Re.NET-MVC-Application-5-of-10h1_ADC3_Eager_loading_example_33f907ff-f0b0-4057-8e75-05a8cacac807.png)
- <span data-ttu-id="839f1-128">*显式加载*。</span><span class="sxs-lookup"><span data-stu-id="839f1-128">*Explicit loading*.</span></span> <span data-ttu-id="839f1-129">这是类似于延迟加载的只不过显式检索相关的数据中的代码;它不会自动发生时您访问导航属性。</span><span class="sxs-lookup"><span data-stu-id="839f1-129">This is similar to lazy loading, except that you explicitly retrieve the related data in code; it doesn't happen automatically when you access a navigation property.</span></span> <span data-ttu-id="839f1-130">加载相关的数据手动通过实体和调用获取对象状态管理器入口`Collection.Load`集合的方法或`Reference.Load`保存单个实体的属性的方法。</span><span class="sxs-lookup"><span data-stu-id="839f1-130">You load related data manually by getting the object state manager entry for an entity and calling the `Collection.Load` method for collections or the `Reference.Load` method for properties that hold a single entity.</span></span> <span data-ttu-id="839f1-131">(在以下示例中，如果你想要加载管理员导航属性，您会取代`Collection(x => x.Courses)`与`Reference(x => x.Administrator)`。)</span><span class="sxs-lookup"><span data-stu-id="839f1-131">(In the following example, if you wanted to load the Administrator navigation property, you'd replace `Collection(x => x.Courses)` with `Reference(x => x.Administrator)`.)</span></span>

<span data-ttu-id="839f1-133">因为它们不立即检索属性值，延迟加载和显式加载也都称为*延迟加载*。</span><span class="sxs-lookup"><span data-stu-id="839f1-133">Because they don't immediately retrieve the property values, lazy loading and explicit loading are also both known as *deferred loading*.</span></span>

<span data-ttu-id="839f1-134">一般情况下，如果您知道你需要相关的数据对于每个实体检索到的预先加载提供最佳性能，因为发送到数据库的单个查询通常比个检索每个实体的单独查询更有效。</span><span class="sxs-lookup"><span data-stu-id="839f1-134">In general, if you know you need related data for every entity retrieved, eager loading offers the best performance, because a single query sent to the database is typically more efficient than separate queries for each entity retrieved.</span></span> <span data-ttu-id="839f1-135">例如，在上面的示例中，假设每个系有十个相关的课程。</span><span class="sxs-lookup"><span data-stu-id="839f1-135">For example, in the above examples, suppose that each department has ten related courses.</span></span> <span data-ttu-id="839f1-136">预先加载示例会导致只需单一 （联接） 查询，将单个往返到数据库。</span><span class="sxs-lookup"><span data-stu-id="839f1-136">The eager loading example would result in just a single (join) query and a single round trip to the database.</span></span> <span data-ttu-id="839f1-137">延迟加载和显式加载示例将同时导致 11 个查询和 11 个往返到数据库。</span><span class="sxs-lookup"><span data-stu-id="839f1-137">The lazy loading and explicit loading examples would both result in eleven queries and eleven round trips to the database.</span></span> <span data-ttu-id="839f1-138">延迟较高时，额外往返数据库对性能尤为不利。</span><span class="sxs-lookup"><span data-stu-id="839f1-138">The extra round trips to the database are especially detrimental to performance when latency is high.</span></span>

<span data-ttu-id="839f1-139">但是，在某些情况下延迟加载是更高效。</span><span class="sxs-lookup"><span data-stu-id="839f1-139">On the other hand, in some scenarios lazy loading is more efficient.</span></span> <span data-ttu-id="839f1-140">预先加载可能会导致非常复杂的联接为其生成 SQL Server 无法有效地处理它。</span><span class="sxs-lookup"><span data-stu-id="839f1-140">Eager loading might cause a very complex join to be generated, which SQL Server can't process efficiently.</span></span> <span data-ttu-id="839f1-141">或者，如果您需要访问实体的导航属性仅对一组实体的子集进行处理，延迟加载可能会更好地执行，因为预先加载将检索数据超过所需的数据。</span><span class="sxs-lookup"><span data-stu-id="839f1-141">Or if you need to access an entity's navigation properties only for a subset of a set of entities you're processing, lazy loading might perform better because eager loading would retrieve more data than you need.</span></span> <span data-ttu-id="839f1-142">如果看重性能，那么最好测试两种方式的性能，以便做出最佳选择。</span><span class="sxs-lookup"><span data-stu-id="839f1-142">If performance is critical, it's best to test performance both ways in order to make the best choice.</span></span>

<span data-ttu-id="839f1-143">通常会使用显式加载，仅当你已启用延迟加载关闭时。</span><span class="sxs-lookup"><span data-stu-id="839f1-143">Typically you'd use explicit loading only when you've turned lazy loading off.</span></span> <span data-ttu-id="839f1-144">你应打开延迟加载关闭时的一个方案是在序列化过程。</span><span class="sxs-lookup"><span data-stu-id="839f1-144">One scenario when you should turn lazy loading off is during serialization.</span></span> <span data-ttu-id="839f1-145">延迟加载和序列化，不要混合，如果您不小心最终查询更多的数据比预期时延迟加载处于启用状态。</span><span class="sxs-lookup"><span data-stu-id="839f1-145">Lazy loading and serialization don't mix well, and if you aren't careful you can end up querying significantly more data than you intended when lazy loading is enabled.</span></span> <span data-ttu-id="839f1-146">序列化通常可通过访问类型的实例上每个属性。</span><span class="sxs-lookup"><span data-stu-id="839f1-146">Serialization generally works by accessing each property on an instance of a type.</span></span> <span data-ttu-id="839f1-147">属性访问触发延迟加载，并为这些延迟加载的实体进行序列化。</span><span class="sxs-lookup"><span data-stu-id="839f1-147">Property access triggers lazy loading, and those lazy loaded entities are serialized.</span></span> <span data-ttu-id="839f1-148">然后，序列化过程访问的延迟加载的实体，可能会导致更多的延迟加载和序列化的每个属性。</span><span class="sxs-lookup"><span data-stu-id="839f1-148">The serialization process then accesses each property of the lazy-loaded entities, potentially causing even more lazy loading and serialization.</span></span> <span data-ttu-id="839f1-149">若要防止此用尽链式反应，启用延迟加载关闭之前序列化实体。</span><span class="sxs-lookup"><span data-stu-id="839f1-149">To prevent this run-away chain reaction, turn lazy loading off before you serialize an entity.</span></span>
